Closure & Exit — complete customer journeys, with what to build

Each tab is one continuous customer journey, screen by screen, in the order a real user moves through it. Existing live screens sit in their real positions; new and improved screens are woven in where they belong. Every screen is tagged Keep / Modify / Add new, with the live Kotlin file noted — so you can read each flow top-to-bottom and see exactly what to add or change in the codebase. The "What to build" box at the top of each flow is the implementation delta.

Grounded: Closure OS v2.0 (16 Jun) · ESR v2.3 · SPR v2.11 §6 · RR v2.3 · live app = wiom-tech/customer-app-kotlin

How to read these flows

This prototype is a build map. Pick a tab, read the phone screens left → right: that is the full customer journey for that exit scenario. The arrows between screens are the transitions. Each screen carries a tag and the live-code reference:

Keep already in the live app — reuse as-is Modify exists, but must change (change noted on the screen) Add new not in the codebase — build it Continuity no closure — service continues

The green "What to build" box at the top of each flow summarises the delta: every Add / Modify item with the screen and the live Kotlin file it maps to.

Why this matters — ground truth

The exit zone is the least-specified part of the Customer OS today, and the most damaging when it fails (FS-02): 38.5% of pickup tickets are silent unrecovered devices · 27% of pickups happen but are never marked · 45.7% refund success · 524 orders stuck at "device taken, money held". Closure OS v2.0 runs two orthogonal machines — relationship (what is owed) × custody (where the device is) — so the system can never lose a device or a refund. CLOSURE_FINAL needs all five conditions: custody resolved · refund resolved (to Stage 3) · both bookend comms · trust signal (if non-return) · obligations resolved.

What the live app has today

Only: a transactional cancel-reason list (CancelBookingReasonScreen.kt), router self-drop/pickup + UPI refund (RouterRecovery*Screen.kt), a payment history that filters refunds out (PaymentHistoryScreen.kt), and a GRACE status with no recall comms. It has nothing for: refund-stage tracking, pro-rata, reversal / welcome-back, custody dispute, bond write-off, SD-first recall, life-event, or CSP-migration continuity. Those are the Add-new screens across the tabs.

Two journeys

4A — self-initiated & reason-driven (8 tabs): the canonical machine, nine reasons converging on CLOSURE_INITIATED → CLOSURE_FINAL or reversing to ACTIVE_ENTITLED. 4B — Wiom-initiated CSP migration (3 tabs): a successful migration is CONNECTION_CHANGED continuity, not closure; only no-alternative + customer-declines reach closure (full refund, trust NONE).

Meena · non-reaffirmation. Plan expires → 15-day reservation (Zone 1, calm) → SD-first recall → no response by T+25 → CLOSURE_INITIATED(non-reaffirmation) → device recovered → deposit auto-refund → CLOSURE_FINAL. Both bookend comms, multi-channel.
