Booking → Install OS: Current App vs New OS

App: customer-app-kotlin  ·  Spec: wiom-cust-b2i-os-v3.1-locked-20260519  ·  Updated 2026-05-29

How the existing Kotlin customer app handles the journey from booking to a live connection, measured against what the new Booking-to-Install (B2I) OS requires. The OS exists to make the journey transparent and fair — so every customer always knows what's happening, and never pays for Wiom's mistakes.

compare_arrowsThe core mismatch

The app treats payment as the finish line and the journey afterwards as a linear chat script. The OS treats payment (booking_fee_paid) as the starting gun — the entry into a governed 16-state machine where every state has an owner and a clock. The app has essentially one status variable (installationStatus: PENDING / INSTALLED / RENEWAL) and a 7-value sticky banner. Everything between "slot picked" and "WiFi connected" — the exact window where real customers get stuck — is invisible in the app.

CURRENT APP location → cost → OTP → PAYMENT → SSID → [dumps to chat] → address → slot → connect WiFi → done │ (treated as the finish line) NEW B2I OS booking_fee_paid ──────────────────────────────────► INSTALL DONE (the starting gun: 16 governed states, each with an owner + a clock)

One-line takeaway

The app has built the pre-payment funnel and the final "it works!" celebration beautifully — but the coordination middle (who's home, finding a partner, slot confirmation, technician on the way, what went wrong) is where the OS lives, and where the app is mostly silent.

priority_highThe 5 biggest gaps to improve

1No install-contact capture (S2)CRITICAL

The OS makes "who will be home?" a hard dispatch prerequisite (INV-B2I-02). The app assumes the booker is always home — the single biggest cause of failed installs. Also ties into the known multi-account identity gap.

Fix → A new screen after address: "सेटअप के समय घर पर कौन होगा?" (same as me / someone else + phone). Block dispatch until resolved.
2No status / tracking surface (S3, S9, S10, S11)CRITICAL

70% of chat is "when will it happen?". The app's only answer is a sticky banner that jumps from "Setup pending" straight to "Live" — it never shows finding partner → slot confirmed → technician assigned → on the way → installing. The OS requires the customer always knows the state (Constitution rule C4).

Fix → A real order-tracking screen showing the live state, the next step, and an ETA.
3No anti-limbo clocks or escalation (S6, S7)CRITICAL

The app has zero timeouts after booking. A silent customer or a non-responsive CSP leaves the booking in limbo forever. The OS says every state has a clock (INV-B2I-07); silence triggers WhatsApp → IVR → Push, then an honest closure-with-refund — never silent consent.

Fix → Server-driven timeouts + escalation ladder; app needs surfaces to reflect "we're trying to reach you" and "respond or we refund".
4No technician visibility (S9 / S10)HIGH

The customer never sees technician name, ETA, or "on the way". The OS makes this a required notification — and it's the trust-builder that actually makes a customer "go home happy".

Fix → Technician card (name + photo + ETA window) plus an en-route notification.
5Drops have no structured attribution (S13)HIGH

The cancel flow collects a reason but has no customer / CSP / system bucket. The OS forbids "unknown" (INV-B2I-04) because attribution is what drives fair refunds — the customer never pays for Wiom's mistakes (Constitution rule C2).

Fix → Structured outcome model + honest messaging: "This one was on us — full refund on the way."

OS #2 · Recharge / Continuity

Recharge / Continuity OS: Current App vs New OS

App: customer-app-kotlin (feature/renewal)  ·  Spec: wiom-cust-continuity-os-v1.0.1-locked-20260528  ·  Updated 2026-05-29

How the existing app keeps an active customer's connection running — recharges, plan renewal, expiry, grace — measured against the new Continuity OS. The OS's whole reason to exist: "It handles customers correctly. It does not manufacture recharges." Non-payment is treated as a neutral, reversible pause — never as churn, never with pressure.

compare_arrowsThe core mismatch

Both sides agree the customer is PayG / recharge-based (the app has even built the forced ₹500→recharge migration). But they diverge on what non-payment means. The app treats expiry as a problem to fix: validity ends → GRACE → net blocked → "Recharge" nudge → if you don't pay, the router gets recalled (ROUTER_REFUND_ORDER_*) and service is stopped. The OS treats non-payment as a neutral, reversible pause: a recharge is a voluntary reaffirmation, absence of one licenses no inference about the customer, and a 15-day reservation window protects the connection with zero coercion.

CURRENT APP ACTIVE → (validity ends) → GRACE → net blocked + "Recharge!" → no pay → ROUTER RECALL → SERVICE STOPPED │ one posture: nudge to pay │ lapse treated as churn/exit NEW CONTINUITY OS ACTIVE → (validity ends) → RESERVATION WINDOW (T+0…T+15) → 8 postures chosen by signals → reaffirm anytime → instant restart │ net off, but relationship alive, support intact, NO pressure, fully reversible to T+25
vpn_key
One-line takeaway

The app has the recharge plumbing (plans, Juspay/QR, autopay, migration, receipts) — genuinely solid. What it lacks is the continuity doctrine: a reservation window instead of a recall, posture instead of one-size nudging, and the hard rule that silence is never read as churn and never met with pressure.

priority_highThe 5 biggest gaps to improve

1Lapse is treated as churn, not a reversible pauseCRITICAL

The app's path is expiry → grace → ROUTER_REFUND_ORDER_* → SERVICE_STOPPED. The OS's core doctrine (INV-CON-01, INV-CON-13) is that absence of recharge is neutral and "lapsed customer" as a classification is prohibited. Reservation expiry is a custody event, never an exit.

Fix → Replace the grace→recall framing with a 15-day reservation window + a T+15→T+25 reversal window. Keep app, support, and recharge fully accessible throughout.
2No posture engine — one nudge for everyoneCRITICAL

The app shows the same "Recharge" CTA whether the net is broken (Wiom's fault), the customer explicitly paused, or quality is degraded. The OS selects 1 of 8 postures from SQ / Recovery / Relationship-State signals. Most importantly: never ask a customer to pay for broken internet (Recovery-first).

Fix → Drive the recharge surface from a server-decided posture (CONTINUITY_POSTURE_DECIDED). At minimum wire Recovery-first + Passive-only before nudging.
3No "pause" concept (Passive-only posture)HIGH

A customer who deliberately wants to pause (travel, seasonal) has no path — the app keeps nudging. The OS's Passive-only posture goes completely silent while keeping the recharge link live, and reads the pause as legitimate, not churn.

Fix → Add an explicit "pause my net" path that silences reminders and preserves a one-tap restart.
4Pre-expiry reminders are skeletalHIGH

There's a MorningReminderScreen but nothing triggers it; NotificationEventScheduler only handles booking, not renewal. No "3 days left" countdown, no renewal WorkManager job — reminders rely entirely on backend CleverTap pushes. The OS needs disciplined, cadence-bounded, non-coercive reminders (CP-CON-05/06).

Fix → Add a renewal reminder worker + an in-app days-left surface, with cadence caps and messaging discipline (no guilt/urgency framing).
5Messaging risks coercion; no anti-pressure guardrailHIGH

The blocked-net "Recharge" banner is the only lever, and there's no guardrail against recharge-under-pressure. The OS bans guilt framing, urgency pressure, and abnormalizing pause across all postures (INV-CON-11), and actively measures a "recharge-under-pressure" guardrail KRA — if it rises, cadence must relax.

Fix → Audit all recharge/expiry copy against the messaging-discipline rules; instrument a pressure guardrail metric.

OS #3 · Service Quality Driver

Service Quality (SQ) Driver OS: Current App vs New OS

App: customer-app-kotlin  ·  Spec: wiom-cust-sq-driver-os-v1.0-locked-20260519  ·  Updated 2026-05-29

How the existing app senses and represents whether a customer's internet is actually working, measured against the new SQ Driver OS. The OS is an epistemic engine — its only job is to emit honest, decomposed quality truth per connection. Its one-line lock: "SQ Driver emits truth. It does not fix internet." This is the foundation every other OS reads from.

compare_arrowsThe core mismatch

This is the starkest gap of the three OSes so far. The app has almost no quality-truth layer at all. "Quality" is a single binary backend flag (NET_ACTIVE / INTERNET_INACTIVE); the only "speed test" is a video the customer watches and self-rates ("did it play smoothly?"); and when net is down, the app just captures a complaint and routes it to a call center. The OS, by contrast, is a continuous epistemic system: it ingests router/netbox telemetry, classifies one of 5 quality states with a confidence score, attributes the fault to one of 8 domains (Wiom-side vs customer-side), and emits that decomposed truth for Continuity, Recovery, and Relationship-State OS to act on.

CURRENT APP net works? → binary flag (active/inactive) ·· "speed test" = a video you self-rate ·· net down → make a ticket → call centre no telemetry · no measurement · no fault attribution · no confidence · ~1050 calls/day NEW SQ DRIVER OS telemetry + complaints → 5 quality states × confidence × 8 fault-domains → QUALITY_STATE_CHANGED → Continuity / Recovery / Relationship-State decomposed truth · "Unknown" is honest · attribution-not-blame · the canonical quality layer everything else reads
vpn_key
One-line takeaway

The app is a thin complaint-capture UI — all quality intelligence is absent or backend/CSP-hidden. The OS makes quality a first-class, canonical, decomposed truth. Without this layer, Continuity's "Recovery-first" posture and Recovery's proactive cases are blind — which is exactly why net-down today collapses into ~1050 calls/day.

priority_highThe 5 biggest gaps to improve

1No quality-truth layer at all — only a paid/unpaid flagCRITICAL

The app's "Live" status means paid, not working. A customer with terrible speed still shows NET_ACTIVE. The OS exists precisely to make quality a canonical, observed truth (C1). Every downstream OS depends on it — without it, "never ask a customer to pay for broken net" is unenforceable.

Fix → Stand up the SQ telemetry pipeline + emit QUALITY_STATE_CHANGED; surface a real quality state in the app distinct from the recharge flag.
2Net-down deflects to ~1050 calls/day instead of self-resolvingCRITICAL

When net is down/slow the app captures a ticket and points the customer at the call centre — no diagnosis, no status, no "we already know your area is down". The OS's cluster-degradation + attribution would let the app proactively say "we see it, we're on it," deflecting the call entirely (the known ~1050 calls/day, ~₹30 each).

It's a video the customer watches and self-rates. It can't distinguish Degraded from Healthy, produces no metric, and only runs once at install. The OS needs measured throughput against a baseline/floor.

4No fault attribution → C2 can't be honoredHIGH

The app can't tell a Wiom network fault from the customer's router placement or device. The OS's 8-domain attribution is what lets the system own its own failures (C2) and route customer-side issues to education instead of a truck-roll.

Fix → Surface FAULT_DOMAIN-driven guidance: Wiom-side → "on us, being fixed"; customer-side → in-app troubleshooting (placement, restart).

Once a ticket is made, the customer sees nothing more and closure is assumed to mean "fixed". The OS rule INV-SQ-16 is explicit: ticket closure ≠ quality restoration — only sustained healthy telemetry counts, with a restoration-window check.

Fix → Show live ticket/restoration status; mark "resolved" only after the SQ restoration window confirms sustained healthy telemetry.

At-Risk Resolution OS: Current App vs New OS

Its one-line lock: "Act on the problem, never on the decision." A customer who leaves with their deterioration fully resolved is counted as a success.

compare_arrowsThe core mismatch

This is an inverse gap from the others. It's not that the app does a primitive version of this OS — it's that the app has no at-risk concept at all, while the OS is an elaborate machine specifically built to prevent the manipulative "retention" the app might naïvely add. Today: everyone is treated identically; a cancellation reason is captured and (for some reasons) routed to a support call — exactly the "save desk" path the OS prohibits; a lapsed customer just gets router recovery, no re-engagement; and a 1-star rating triggers nothing. The OS instead: consumes an evidence-backed AT_RISK signal, opens a governed case, and resolves the underlying problem with a bounded 8-tool kit — never a discount, never a "please stay."

CURRENT APP unhappy / leaving → capture reason → (some) route to support call ·· lapsed → router recall ·· 1★ rating → nothing no at-risk concept · everyone treated the same · reason logged, not actioned · the "call" can become an ad-hoc save desk NEW ARR OS RS OS emits AT_RISK (evidence-backed) → governed case → resolve the PROBLEM via T1–T8 → free decision "act on the problem, never the decision" · non-contingent compensation · exit-with-resolution = success · NO save offers
vpn_key
One-line takeaway

The right move here is not "add retention features." The OS's whole point is that a save desk is the wrong answer. The app needs to (a) gain an evidence-backed at-risk concept fed from Relationship-State OS, and (b) retire the ad-hoc reason→call→improvise path in favour of governed problem-resolution. The app's biggest risk isn't doing too little — it's that its support-call path can quietly become the coercive save desk the OS exists to forbid.

compare_arrowsThe core mismatch

The app's exit is an implicit, logistics-only drift: there's no "leave" button — a customer lapses → GRACE → SERVICE_STOPPED, and in parallel the router-recovery + deposit-refund flow runs (self-drop vs pickup, UPI). It's reasonably clean (no save-desk obstruction), but it collapses relationship and custody into one path, has no real reversibility once stopped, no closure/refund-status surface, and wipes all memory on logout. The OS separates relationship-side (EXIT_INITIATED → EXITED_FINAL) from custody-side (device states), makes exit frictionlessly reversible until physical pickup, requires two bookend communications (no silent exit), and preserves person-level memory + obligations after the service instance closes.

One-line takeaway

The app already does the logistics (get the router back, refund the deposit) — but the OS is about integrity at the moment of maximum power asymmetry: never let an exit go silent, keep it reversible, keep the customer oriented on their money, and never let the device/refund land in limbo. The app's real-world pain — 27% of pickups unmarked, 38.5% silent unrecovered devices, 524 orders stuck with money held — is exactly the canonical-custody-truth failure the two-state-machine design fixes.

priority_highThe 5 biggest gaps to improve

1Custody & refund land in limbo — devices and money get stuckCRITICAL

The app blends relationship + custody into one status and has no canonical device-truth: 27% of pickups are never marked, 38.5% of devices go silently unrecovered, and 524 orders sit with the device taken but money held. The OS's separate custody state machine with pickup-proof + a 60-day bond long-stop is precisely the fix.

Fix → Split custody from relationship state; require pickup-proof (netbox match + condition + scan/OTP); add a bond-hold long-stop so nothing sits in limbo.
2Silent exit — no bookend communication, no closureCRITICAL

Service just stops; the customer sees "Service suspended" with no path, no refund timeline, no closure. The OS's No-Silent-Exit rule mandates two multi-channel bookend comms (start + finish) and an SD-first frame: "Your ₹300 deposit is still with Wiom — we don't want your money blocked." Silence is the OS's primary harm target.

Fix → Send a start-of-exit + end-of-exit message (SMS/WhatsApp, not app-only) with refund amount + timeline; show a real closure state.
3No real reversibility once stoppedHIGH

Reversal only works during GRACE; once SERVICE_STOPPED the only path is a full cold rebook. The OS keeps exit reversible right up to physical pickup — recharge during the window instantly cancels the recall (Wiom absorbs the dispatch cost), with zero fee or penalty, because friction-on-return punishes the exact outcome you want.

Fix → Add a recall-cancelled path: recharge any time before pickup → instant reactivation, zero fee, one confirmation.
4No refund-status surface or SLAHIGH

After UPI is submitted the customer sees a state flag but no timeline, no "refund by X", no completion event. The OS makes refund treatment rule-based (self-return ₹300 vs pickup −₹50) with a target refund window the customer is told about (C4 clarity).

Fix → Show refund amount, method-based tier, and a committed timeline; track and surface PROCESSING → COMPLETED.
5Memory wiped on logout — and obligations can vanishHIGH

Logout clears everything; a returning customer starts cold, and anything Wiom owed (a pending refund, a Recovery credit) has no person-level home. The OS closes the service instance but preserves the person, trust signals, and live obligations (INV-EXIT-19: exit never extinguishes obligations) via CM OS.

Fix → Persist person-level history + open obligations across exit/logout; greet returners with context, never drop what Wiom owes.
